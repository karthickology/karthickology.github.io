Experiment #01: adding a low‑quality image on my site & checking performance impact

🎯 MY GOAL

Measure how a single unoptimized, low‑quality (visually blurry & large file size) image affects Core Web Vitals, page load time, and Lighthouse performance score. Identify real user impact.

⚙️ THE PROCESS

1. Baseline audit (Lighthouse, load timings, network tab).
2. Added a 2.8MB JPG image (2400x1600px, quality 30%, no lazy loading) inside hero section.
3. Repeated the same test environment (3 runs, desktop/3G slow simulation).
4. Compared before/after metrics + visual screenshot analysis.

📸 BEFORE · without low‑quality image
Performance: 94
LCP: 1.2s ✅
CLS: 0.02 ✅
Total image weight: 0.22 MB
🖼️ [ baseline preview ]
Fast render
Lighthouse 9.2 · Fully loaded: 1.8s
⚠️ AFTER · with low‑quality image (heavy & unoptimized)
Performance: 68
LCP: 3.9s ❌ (blocked)
CLS: 0.19 ⚠️ (layout shift)
Total image weight: +2.8 MB
🖼️ [ blurry + oversized ]
Render delayed
Lighthouse 6.4 · Fully loaded: 4.7s
📌 OUTPUT & CONCLUSION

✔️ Verdict: Adding the low-quality but heavy image degraded performance by -26 points in Lighthouse, increased LCP by 225%, and introduced visible layout shift. Even though image was “low quality” visually, file size was the real enemy. Lesson: Always compress & use responsive images (WebP/AVIF). Performance matters even for small design changes.

Experiment #02: Lazy loading removal + above‑fold image priority (before/after analytics)

🎯 MY GOAL

Analyze how removing lazy‑loading from above‑fold hero image influences LCP and user experience, compared to a properly lazy‑loaded version.

⚙️ THE PROCESS

Baseline: hero image with lazy loading (standard). Variant: removed `loading="lazy"` + preload hint. Measured using Web Vitals extension & real screenshot comparison from DevTools performance panel.

📸 BEFORE (lazy loading enabled)
Performance: 88
LCP: 2.1s (lazy delayed)
First paint: 1.4s
⏳ [ placeholder → image after scroll ]
📸 AFTER (no lazy + preload)
Performance: 91 ▲ +3
LCP: 1.6s ✅ (faster)
First paint: 1.1s
⚡ [ image loads immediately ]
📌 OUTPUT & CONCLUSION

✔️ Result: Removing lazy loading from critical hero image improved LCP by ~0.5s and boosted overall perceived performance. However, for below‑fold images lazy loading still recommended. Action: selective lazy loading — not a global rule.

Experiment #03: Render‑blocking CSS vs optimized critical CSS – full analytic snapshot

🎯 MY GOAL

Understand how large external CSS files block first paint and affect Lighthouse “Eliminate render‑blocking resources” score. Show detailed before/after metrics.

⚙️ THE PROCESS

Baseline: 78KB of external CSS (render‑blocking). After: inlined critical CSS + deferred non‑critical styles. Ran Lighthouse 6 times, captured screenshots of coverage tab & filmstrip.

LCP: 3.8s
Blocking time: 410ms

✅ Strong improvement: Removing render‑blocking CSS reduced FCP by 58% and boosted performance score from 62 → 93. Why this matters: better user retention, higher conversion. Best practice: always inline critical styles and load rest asynchronously.

🔍 What am I doing & why it matters

Every experiment you see here is based on real-world performance auditing. I deliberately introduce a change (e.g., low‑quality image, lazy loading toggle, CSS optimizations) → measure using analytics tools (Lighthouse, Web Vitals) → compare before/after screenshots. The goal: provide actionable insights for your site. Performance affects SEO ranking, bounce rate, and revenue. Each log includes goal → process → conclusion so you know exactly what was done, how, and what to apply.
